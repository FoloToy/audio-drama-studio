## 0. 一句话定义

**把一部原著或一个主题，变成一个可持续更新、安全合规、成本可控的儿童音频剧系列的生产线。**

[Δ] V1 的定义是"把文本变成一集音频的创作工具"。V2 把定义单位从**「集」升级为「系列」**，把定位从**「工具」升级为「生产线」**。理由：

1. 儿童音频内容的商业形态是**连载系列**（10 集西游记、52 集成语故事），不是单集。V1 的数据模型里项目=一次性任务，没有"第二季再加 10 集"的概念，这在实现中已经暴露为硬伤（角色跨集一致性、素材复用、增量生产都没有承载实体）。
2. 生产线的核心指标是**吞吐、一次通过率、单集成本**，工具的核心指标是"能不能做出来"。MVP 已经证明"能做出来"；V2 要回答"能不能便宜、稳定、安全地批量做"。

## 1. 重新审视：V1 定义的六个盲区

这是本次重新定义的出发点，逐条对应后文的方案。

| # | 盲区 | 后果（已在 MVP 中验证或可预见） | V2 对策 |
|---|------|------|------|
| 1 | **没有"系列"实体** | 角色声音/性格无法跨项目复用；续集要重新识别角色、重新绑定声音 | §5 新增 Series + 角色卡（Character Bible） |
| 2 | **没有版本概念** | 重新生成即覆盖；用户不敢点"重新生成"；无法对比两版剧本/两条录音 | §5 新增 Revision/Take；重新生成默认产生新版本 |
| 3 | **质量只管文本安全，不管音频质量** | 混音后的响度、静音段、时长偏差无人把关；多音字读错无纠正手段 | §8 质量体系：响度标准、质检 Agent、发音词典 |
| 4 | **成本不可见** | BGM 一首 3-4 分钟、LLM 每集数万 token，批量生产时成本失控无预警 | §9 成本账本：每集预算/实际，生成前预估 |
| 5 | **合规只到"儿童安全"** | 遗漏 AIGC 内容标识义务、素材版权来源、声音授权链路 | §7 合规三支柱：儿童安全 + AIGC 标识 + 版权 |
| 6 | **发布是终点断头路** | 导出 MP3 后一切靠手动；而内容的归宿是播客平台和故事机 | §6 发布通道成为一等公民（P1），打通到消费端 |

## 2. 目标用户（重排）

[Δ] V1 按"编辑/编剧/音频制作/管理员"的**职能**划分角色，适合描述权限，不适合描述购买动机。V2 按**客户类型 + JTBD** 重排：

### P0 · 儿童内容工作室 / 出版社数字部门
- **JTBD**：手里有 IP 或公版书目，需要以传统制作 1/10 的成本，稳定地把书目转成音频剧上架。
- **买单理由**：省编剧+配音+后期的人力；批量、可预期的交付。
- **对产品的核心要求**：批量生产、系列一致性、安全兜底（他们对下架/投诉最敏感）、成本报表。

### P1 · 独立创作者 / 儿童播客主
- **JTBD**：一个人维持一档周更儿童故事播客。
- **对产品的核心要求**：单人可完成全流程、一键发布 RSS、单集成本低于其广告/订阅收入。

### P2 · 硬件 / IP 方（故事机、早教机内容供给）
- **JTBD**：为设备端持续供给独家内容库。
- **说明**：仓库中存在 FoloToy 品牌痕迹，推断本产品可能与故事机生态同源。若成立，这是差异化最强的通道——竞品（剪映、通用 TTS 工具）都不直连硬件消费端。**列为开放问题 OQ-1**（§12）。

### 明确非目标用户
- C 端家长（自己给孩子做故事）：需求存在但付费意愿和使用频次都撑不起产品复杂度，V2 不做，未来可用极简模板化外壳覆盖。

## 3. 价值主张与竞争位置

[Δ] V1 无竞争分析。V2 明确回答"为什么不用现成工具拼"：

用户的替代方案是拼工具链：ChatGPT 写稿 → 剪映/TTS 配音 → AU 手工混音 → 手动上传。痛点不在任何单点，而在**流水线本身**：

1. **改编质量**：通用 LLM 不懂"适龄"——5 岁和 10 岁的句长、冲突强度、死亡表达完全不同。我们的适龄规则 + 风格模板 + 安全规则是领域资产。
2. **多角色一致性**：拼工具链下，第 3 集的孙悟空和第 7 集音色/性格漂移。我们用角色卡 + 声音绑定锁定。
3. **安全兜底**：个人凭感觉审核，机构需要可审计的审核记录。我们提供三级审核 + 审核留痕。
4. **结构化资产**：台词、音效、BGM、角色都是结构化数据，支持局部重生成——这是"导出成品 MP3 的工具"做不到的。

**一句话卖点：不是更好的 TTS，而是唯一按"儿童内容生产合规流水线"设计的产品。**

## 4. 设计原则（新增章节）

[Δ] V1 没有原则层，导致功能取舍无依据。V2 确立六条，后文所有 P0/P1 判断都回溯到这里：

1. **人的确认是闸门，不是装饰**。四个硬闸门（大纲、剧本、声音、成片）不可跳过、留痕可审计；其余环节全自动。
2. **安全是硬门禁**。高风险内容在数据层阻断进入音频生成（HTTP 层可绕过 UI，但绕不过 orchestrator 的门禁校验）。
3. **一切生成物皆资产**。音效、BGM、角色卡、风格模板、发音词典默认入库复用；第二次生产同类内容的边际成本必须显著下降。
4. **系列一致性 > 单集精彩度**。冲突时（比如某集重生成会改变角色性格），一致性胜出——锁定机制的存在意义。
5. **成本先可见，再可控**。每次生成前预估、生成后记账；无账本不谈优化。
6. **可回溯、可局部重做**。任何一句台词可以单独改文本、单独重配音、单独替换进成片，不需要全量重跑。

## 5. 领域模型 V2

[Δ] 在已实现的 15 张表基础上，新增/调整以下实体（增量，不推翻）：

```
Series（系列/IP）            ← 新增，一级实体
 ├── CharacterBible（角色卡） ← 新增：跨项目的角色设定+声音绑定+禁用表达
 ├── PronunciationLexicon    ← 新增：系列级发音词典（多音字、人名读音）
 └── Project（季/批次）       ← 原 Project 降为 Series 的子集
      ├── SourceMaterial / SourceAnalysis        （已有）
      ├── Episode
      │    ├── ScriptRevision ← 新增：剧本版本；ScriptBlock 挂在 Revision 下
      │    │    └── ScriptBlock（已有，+revision_id）
      │    ├── AudioTake      ← 新增：同一 block 的多条录音候选
      │    └── FinalMix（audio_assets kind=final，+版本号）
      ├── SafetyFinding + SafetyAuditLog ← 新增审核留痕（谁、何时、如何处理）
      └── CostLedger          ← 新增：每次 Agent 调用的 token/字符/时长/金额
 Channel（发布通道）          ← 新增：RSS/播客平台/故事机设备库
 └── PublishRecord            ← 新增：哪个成片版本发到了哪个通道
```

**兼容策略**：现有单项目数据视为"单项目系列"（Series 与 Project 1:1 自动包裹），零迁移成本。

## 6. 核心流程 V2：两条流水线

[Δ] V1 只定义了冷启动闭环。V2 拆成两条，因为第二条才是生产线的日常：

### 6.1 首季冷启动（≈ V1 闭环，已实现）
```
建系列 → 输入素材 → 解析+安审 → 拆集大纲 →[闸门1]→ 逐集剧本+安审 →[闸门2]
→ 角色识别 → 建角色卡 → 声音绑定 →[闸门3]→ 逐集音频+质检 →[闸门4]→ 发布/导出
```

### 6.2 续集增量生产（新增，P0）
```
选已有系列 → 输入新素材/新章节
→ 复用：角色卡、声音绑定、风格、发音词典、已有音效/BGM 库
→ 只新增：新集大纲 →[闸门1]→ 新集剧本 →[闸门2]→（仅新角色需绑声音）→ 音频 →[闸门4]
```
目标：**续集单集的人工操作 ≤ 3 次确认点击**，生产时长与成本比首季下降 50% 以上（复用资产）。

## 7. 合规三支柱（强化）

[Δ] V1 只有支柱一。机构客户的采购前提是三条都齐：

1. **儿童内容安全**（已实现骨架）：规则引擎（硬阻断，不依赖 LLM）→ LLM 语义审核 → 人工确认闸门；全程 `SafetyAuditLog` 留痕。
2. **AIGC 标识**（新增，P0 合规项）：按《人工智能生成合成内容标识办法》（2025.9 起施行），导出音频需附显式声明（片头/片尾语音或元数据）+ 隐式标识（文件元数据写入生成信息）。做成导出选项，默认开启。
3. **版权与授权**（新增，P1）：素材登记来源类型（公版/已授权/原创），非公版素材要求上传授权声明；声音授权状态已建模（V1 已有），补充"授权到期日"字段。

## 8. 质量体系（新增章节）

[Δ] MVP 实测暴露：文本能出、音频能混，但没有任何"好不好"的判据。V2 建立两层：

### 文本质量（剧本闸门前自动检测，结果附在审阅页）
- 适龄指标：平均句长、生僻字率（对照年龄段词表）、单集角色数上限
- 结构指标：旁白/对话比是否符合所选风格模板的定义
- 一致性指标：角色台词是否违反角色卡的"禁用表达"

### 音频质量（成片闸门前，质检 Agent 自动执行）
- **响度归一**：成片统一到 -16 LUFS（播客标准），峰值 ≤ -1 dBTP —— pydub 管线中加归一化步骤
- **完整性**：逐 block 检测静音/截断/TTS 失败残留；成片时长与预估偏差 >20% 报警
- **发音**：命中系列发音词典的词强制注音重合成（豆包支持拼音标注）；质检页提供"逐句试听 + 标记读错 → 一键入词典 → 重合成"闭环

## 9. 成本模型（新增章节）

每集成本 = LLM tokens（解析+大纲+剧本+审核+提示词）+ TTS 字符 + 音效生成 + BGM 生成 + 存储。

量级估算（8 分钟/集，≈2000 字台词）：LLM 数万 token + TTS 2000 字 + 音效 3-5 个 + BGM 1-2 首。其中 **BGM 是成本和时长的大头**（MiniMax ~250s/首）——这正是素材库复用策略的价值：BGM 库命中率每提高一成，单集生产时长显著下降。

产品要求：
- 生成前展示预估成本/时长；`CostLedger` 记录实际
- 系列报表：成本/集趋势、库命中率
- 配额：为多租户预留 per-user 月度配额字段（P2）

## 11. 指标体系（新增章节）

**北极星：每周由平台产出并通过全部闸门的"可发布分钟数"。**

| 类别 | 指标 | 说明 |
|---|---|---|
| 效率 | 单集端到端生产时长 | 从素材到成片（含人工等待） |
| 效率 | 首季 vs 续集成本比 | 验证资产复用价值，目标 ≤ 0.5 |
| 质量 | 剧本一次通过率 | 闸门2 无修改直接确认的比例 |
| 质量 | 单句重生成率 | 越低说明 TTS/情感标注越准 |
| 安全 | 高风险拦截数 / 漏报数 | 漏报=人工在后置环节才发现的风险 |
| 资产 | BGM/音效库命中率 | 复用 vs 新生成 |

## 12. 已决策事项与开放问题

**已决策（2026-07）：**

- **D-1 设备内容库发布 = P0**（原 OQ-1）：成片直接发布到自有设备（故事机）内容库，是本产品最强差异化通道。发布通道抽象为 Channel，首个实现为 `device_library`（可配置 API 地址 + 鉴权 Key）。
- **D-2 全能力多供应商 = P0**（原 OQ-3 扩大）：**LLM、TTS、音乐（BGM）、音效、图片生成五种能力全部支持多供应商**。系统设置中可同时配置多家供应商的 Key；生成文本 / 构建音色 / 生成音乐音效图片时可按项目选择供应商（未选则用全局默认）。声音建模为 `voice = provider + provider_voice_id`。
- **D-3 新增图片生成域 = P0**（新增能力）：角色头像、故事集（项目/系列）封面等由图片生成供应商产出，纳入同一供应商抽象与资产库体系。

**供应商矩阵（首发适配）：**

| 能力 | 供应商 | 说明 |
|---|---|---|
| LLM | DeepSeek / OpenAI 兼容（可自定义 Base URL，覆盖 GPT、Qwen、Moonshot 等）/ Anthropic（含 OpenRouter） | 项目级可选 |
| TTS | 豆包 seed-tts-2.0 / MiniMax speech / ElevenLabs multilingual | 音色带 provider 标识，同一剧可混用 |
| 音乐 BGM | MiniMax music-2.6 / Suno（本地代理） | |
| 音效 SFX | ElevenLabs Sound Effects / MiniMax | |
| 图片 | OpenAI 兼容（gpt-image / DALL·E 及兼容代理）/ 火山方舟 Seedream | 角色头像、封面 |

**仍开放：**

- **OQ-2 商业模式**：SaaS 订阅+用量计费，还是内部生产工具？决定多租户/配额的优先级。
- **OQ-4 人工配音混合**：机构客户可能要求主角色真人配音+AI 配角。AudioTake 模型天然支持"上传的 take"，是否开放为功能？
- **OQ-5 设备库接口规范**：设备内容库的上传 API 契约（字段、鉴权、回调）需硬件侧提供；当前按"可配置 URL + Bearer Key + multipart 上传"实现，后续对齐。

## 13. 账号与工作区域（V2.1 完整性审查补充）

> 背景：V2 初稿聚焦生产线域，遗漏了"平台作为多人使用的后台"所需的账号与工作区闭环。本节按一次完整性审查的结果补充定义，并确立一条**导航完整性原则**。

### 13.1 账号闭环（P0，已实现开发版）

| 项 | 定义 | 现状 |
|---|---|---|
| 登录 | 邮箱即账号（`POST /api/login` upsert users 表），开发模式不校验密码；生产化时替换为密码/SSO，接口不变 | ✅ |
| 退出 | 两处入口：侧栏底部用户卡的退出图标；工作台右上头像菜单「退出登录」。退出清除本地会话回到登录页 | ✅ |
| 用户菜单 | 头像 ▾ 下拉：用户名/邮箱、系统设置、退出登录 | ✅ |
| 身份贯穿 | `projects.created_by` = 登录用户；为后续"只看自己的项目"（权限规则 PRD §11）铺路 | ✅ |
| 角色 | user / admin（users.role）；管理员标识显示在侧栏用户卡 | ✅（未做页面级拦截，P1） |

### 13.2 导航完整性原则

**平台上每一个可点击的入口必须有真实的目的地与数据来源，禁止装饰性 UI。** 本次审查修复清单：

| 入口 | 修复前 | 修复后 |
|---|---|---|
| 通知铃铛 | 假红点、点击无反应 | 任务通知下拉（最近 Agent 任务 + 状态），红点=存在进行中任务，直达任务中心 |
| 我的资源 | 落到素材库（错误目的地） | 独立页：生成图片墙 / 成片音频（可试听下载）/ 导出+发布记录时间线 |
| 项目名 ✎ | 死图标 | 重命名项目 |
| 保存草稿 | 只弹提示 | 实际落库（touch updated_at） |
| 使用指南/帮助中心 | 死链 | 使用指南弹窗（五步流程 + Key 配置引导） |

### 13.3 项目操作完备性（对齐 PRD §8.1）

### 13.4 状态聚合规则（修复）

项目状态是**分集状态的聚合**，不是最后一次操作的副作用：

- `script_approved` ⇐ 当且仅当**全部**分集 review_status=approved；部分通过时保持 `script_review`（本次修复的 bug）。
- `completed` ⇐ 全部分集 audio_status=generated（当前实现为单集完成即置位，待下轮对齐，P1）。
- 聚合规则统一收敛到后端，前端只读。
